# 2. 맵과 로봇을 코드로 표현하기 — Cell, Map, Agent

[01_mapf_problem.md](01_mapf_problem.md)에서 정의한 개념(칸, 시각, 충돌)을 실제로
C++ 코드에서 어떻게 다룰지 정합니다. 이 장은 알고리즘이 아니라 **자료구조** 설계입니다.
알고리즘(A*, PBS)을 짜기 전에 "데이터를 어떻게 담을지"부터 확실히 해야, 나중에
알고리즘 코드가 훨씬 간단해집니다.

## 2.1 Cell — 격자 위의 한 칸

가장 작은 단위입니다. 그냥 (x, y) 좌표 한 쌍입니다.

```cpp
struct Cell {
    int x{0};
    int y{0};
};
```

규칙:
- x = 열(가로), y = 행(세로).
- 왼쪽 위가 (0, 0). 오른쪽으로 갈수록 x 증가, 아래로 갈수록 y 증가.
  (화면 좌표계와 동일하게 맞춰서, 나중에 시각화(pygame)할 때 헷갈리지 않게 합니다.)

비교 연산자(`==`, `<`)가 필요한 이유: 나중에 "이 칸이 저 칸과 같은가"를 비교하거나,
`std::map`/`std::set`에 Cell을 키로 넣어 정렬해야 하는 경우가 생깁니다. 처음부터
정의해두면 매번 새로 만들 필요가 없습니다.

## 2.2 SpaceTimeCell — 칸 + 시각

[01장 1.4절](01_mapf_problem.md)에서 말한 "시공간(space-time)" 상태를 담는 자료구조입니다.

```cpp
struct SpaceTimeCell {
    int x{0};
    int y{0};
    int t{0};   // 타임스텝
};
```

"3번째 타임스텝에 (2,1) 칸에 있다" = `SpaceTimeCell{2, 1, 3}`.
Space-time A*가 탐색하는 모든 상태가 이 타입입니다.

## 2.3 Map — 격자 전체

맵은 "가로×세로 크기"와 "어느 칸이 벽인지" 정보를 담습니다.

```cpp
class Map {
public:
    Map(int width, int height);                       // 빈 맵 생성
    explicit Map(const std::vector<std::string>& rows); // ASCII로 그리기
    static Map from_file(const std::string& path);     // 표준 벤치마크 포맷 읽기

    bool is_passable(int x, int y) const;               // 지나갈 수 있는 칸인가?
    std::vector<Cell> neighbors(Cell c) const;           // 이동 가능한 다음 칸들
    void set_obstacle(int x, int y);                     // 런타임에 벽 추가
};
```

### 왜 생성 방법이 3가지나 되는가

| 생성 방법 | 언제 쓰는가 |
|---|---|
| `Map(width, height)` | 테스트에서 "그냥 빈 공간"이 필요할 때. 가장 단순. |
| `Map({"#####", "#...#", ...})` | 테스트에서 **특정 모양의 좁은 통로**를 일부러 만들 때. 문자열을 보면 맵 모양이 눈에 바로 보여서 디버깅이 쉽습니다. (`'#'` = 벽, `'.'`/`' '` = 빈 칸) |
| `Map::from_file(path)` | 논문 실험에서 표준 벤치마크 맵(MovingAI 포맷)을 불러올 때. |

### neighbors가 "대기(wait)"를 포함하는 이유 — 중요

`neighbors(c)`는 상/하/좌/우 4방향뿐 아니라 **제자리에 머무는 것도 후보로 포함**합니다.
처음 보면 "왜 안 움직이는 것까지 옵션이지?"라고 생각하기 쉽지만, 이게 없으면
시공간 A*가 풀 수 있는 문제의 범위가 크게 줄어듭니다.

```
예: 좁은 통로에 로봇 B가 막고 있고, 1턴 후에 비켜준다면.
   로봇 A가 "대기" 없이는 — 무조건 전진/후퇴만 해야 하므로 이 상황을 표현할 수 없음.
   "대기"가 있으면 — A가 1턴 기다렸다가 B가 비킨 후 지나간다, 라는 해를 찾을 수 있음.
```

즉 "대기"는 MAPF에서 거의 항상 필수입니다. 04장(Space-time A*)에서 이게 실제로
탐색 트리에 어떻게 들어가는지 봅니다.

### set_obstacle — 정적 장애물 vs 동적 장애물 구분이 중요

`set_obstacle(x, y)`는 맵에 "영원히 막힌 벽"을 하나 추가합니다. 주의할 점:
이 함수는 **정적 장애물**(한번 생기면 계속 거기 있는 장애물, 예: 떨어진 화물)만 다룹니다.

"동적 장애물"(시간이 지나면 사라지거나 이동하는 장애물, 예: 다른 로봇, 사람)은
Map이 아니라 [03_reservation_table.md](03_reservation_table.md)의 ReservationTable이
"시각 t에만 막혀있다"는 형태로 다룹니다. 이 구분을 헷갈리면 나중에
"장애물이 사라졌는데 왜 계속 막혀있지?" 같은 디버깅 지옥에 빠집니다 — 지금부터
정확히 나눠서 기억하세요.

| | 누가 관리 | 언제까지 막혀있나 |
|---|---|---|
| 정적 장애물 | `Map::set_obstacle()` | 영원히 (다시 풀어주는 함수 없음) |
| 동적 장애물(다른 로봇 등) | `ReservationTable` | 정해진 시각(t)에만 |

## 2.4 Agent — 로봇 한 대

```cpp
struct Agent {
    int id{0};       // 로봇 번호 (0, 1, 2, ...)
    Cell start{};    // 출발 칸
    Cell goal{};     // 목적지 칸
};
```

아주 단순합니다. 로봇을 표현하는 데 필요한 최소 정보(번호, 출발지, 목적지)만 담습니다.

`id`가 왜 필요한가: PBS(05장)는 여러 로봇의 경로를 한꺼번에 다루는데,
결과를 "로봇 id → 경로" 형태의 표로 돌려줍니다. 또한 06장의 "계층적 확장"에서는
"어느 로봇이 길을 막고 있는지" 추적해야 하므로, 모든 칸 점유 정보에 **누구 것인지**
번호가 붙어 있어야 합니다.

## 2.5 우선순위는 어디에 있는가? — "벡터의 순서가 곧 우선순위"

여기서 헷갈리기 쉬운 부분입니다. `Agent` 구조체 자체에는 "우선순위" 필드가
없습니다. 대신, PBS에게 넘기는 `std::vector<Agent>`에서 **앞에 있는 로봇일수록
우선순위가 높다**는 규칙을 씁니다.

```cpp
std::vector<Agent> agents = { agentA, agentB, agentC };
// agentA가 1순위, agentB가 2순위, agentC가 3순위.
```

왜 따로 우선순위 필드를 안 만들고 "벡터 순서"로 정했는가: 단순함을 위해서입니다.
순서를 바꾸고 싶으면 벡터를 정렬(`std::sort`)하기만 하면 되고, Agent 구조체를
건드릴 필요가 없습니다. 05장(PBS)에서 이 "순서 = 우선순위" 규칙이 실제로
충돌을 막는 데 어떻게 쓰이는지 봅니다.
