# 3. 예약 테이블 — 충돌을 막는 장부

[01장](01_mapf_problem.md)에서 충돌의 정의(vertex/edge conflict)를 정했고,
[02장](02_map_and_agent.md)에서 칸·시각·로봇을 코드로 표현하는 법을 정했습니다.
이제 "여러 로봇이 서로의 자리를 침범하지 않도록, 누가 언제 어디 있는지 기록하는
장부"를 만듭니다. 이게 **ReservationTable**입니다.

## 3.1 비유: 식당 예약 장부

식당에 테이블이 여러 개 있고, 손님들이 시간대별로 자리를 예약한다고 생각하세요.

- **칸(Cell)** = 테이블 자리
- **시각(t)** = 예약 시간대 (1번째 턴, 2번째 턴, ...)
- **예약 테이블(ReservationTable)** = "어느 자리가 몇 시에 누구에게 예약되어 있는지" 적은 장부

장부에 "1시에 3번 테이블, 김씨 예약"이라고 적혀 있으면, 다른 손님은 1시에 3번 테이블을
못 씁니다. MAPF에서도 똑같습니다 — 한 로봇의 경로가 정해지면, 그 경로가 차지하는
모든 (칸, 시각)을 장부에 적어두고, 다음 로봇은 그 장부를 보고 빈 자리만 골라서
자기 경로를 찾습니다.

## 3.2 자료구조: 무엇을 저장하는가

```cpp
class ReservationTable {
public:
    void reserve(int x, int y, int t, int agent_id = -1);
    void unreserve(int x, int y, int t);
    bool is_occupied(int x, int y, int t) const;
    std::optional<int> get_owner(int x, int y, int t) const;

    void reserve_edge(int x1, int y1, int x2, int y2, int t);
    bool is_edge_occupied(int x1, int y1, int x2, int y2, int t) const;

    void clear();
};
```

내부적으로는 두 개의 저장소를 갖습니다.

| 저장소 | 키 | 값 | 막는 충돌 종류 |
|---|---|---|---|
| `vertices_` | (x, y, t) | 예약한 로봇의 id | Vertex conflict |
| `edges_` | (x1,y1,x2,y2,t) | (존재 여부만) | Edge conflict |

### Vertex 예약 — `reserve(x, y, t, agent_id)`

"타임스텝 t에 칸 (x,y)는 agent_id가 차지한다"를 기록합니다.
`is_occupied(x, y, t)`로 "이미 누가 차지했는가?"를 확인할 수 있습니다.

### Edge 예약 — `reserve_edge(x1, y1, x2, y2, t)`

[01장 1.3절](01_mapf_problem.md)의 edge conflict(자리 바꿈)를 막기 위한 예약입니다.
"타임스텝 t→t+1 사이에 (x1,y1)→(x2,y2) 이동을 한다"를 등록하면, **역방향**
(x2,y2)→(x1,y1) 이동도 자동으로 같이 차단됩니다. 그래야 두 로봇이 서로 마주보고
스쳐 지나가는 걸 막을 수 있습니다 — 한쪽만 막으면 반대쪽에서 들어오는 로봇을
못 잡습니다.

## 3.3 핵심 설계 결정: "점유했다"가 아니라 "**누가** 점유했다"

가장 단순한 구현이라면 `vertices_`를 `unordered_set<(x,y,t)>`로 만들어서
"있다/없다"만 기록하면 충분합니다. 실제로 이 프로젝트도 처음엔 그렇게 시작했습니다.

그런데 이렇게 하면 한 가지를 답할 수 없습니다.

> "이 칸이 막혀서 못 지나갔다. **그런데 누가 막고 있었던 거지?**"

이 질문이 왜 중요한지는 [06_selective_replan.md](06_selective_replan.md)에서
"계층적 확장(Tiered Escalation)"을 다룰 때 결정적으로 중요해집니다. 미리 한 줄로
말하면: 재계획이 실패했을 때 "막은 원인 로봇"을 찾아서 그 로봇까지 같이 재계획에
끌고 와야 하는데, 장부에 "누구 예약인지"가 없으면 그 원인 로봇을 찾을 방법이
없습니다.

그래서 `vertices_`는 `unordered_set` 대신 **`unordered_map<(x,y,t), agent_id>`**
입니다 — 자리가 찼는지뿐 아니라, **누가** 그 자리를 예약했는지까지 항상 같이
기록합니다. `get_owner(x, y, t)`로 그 주인을 조회할 수 있습니다.

```
unordered_set<(x,y,t)>             "여기 누가 있다" (있다/없다만)
unordered_map<(x,y,t), agent_id>   "여기 누가 있다 + 그게 누구다"   ← 이 프로젝트가 쓰는 것
```

`agent_id`의 기본값이 -1인 이유: 가끔 "예약은 해야 하는데 누구 것인지 추적할
필요가 없는" 경우가 있습니다(예: 정적 장애물을 임시로 표시할 때). 이런 경우엔
-1("주인 모름")로 등록합니다. -1이어도 "점유는 되어 있다"는 사실 자체는
정확히 유지됩니다 — `get_owner`가 -1을 반환하는 것과, 아예 점유가 안 되어
있어서 `std::nullopt`를 반환하는 것은 구분됩니다.

## 3.4 unreserve가 필요한 이유 — "목적지 임시 선점"

PBS(05장)는 로봇 하나의 경로를 계획하기 전에, 한 가지 안전장치를 둡니다:
"아직 차례가 안 된 다른 모든 로봇들의 **목적지**를 임시로 선점해 둔다." 이렇게 하면
지금 계획 중인 로봇이 "다른 로봇의 최종 도착 자리"를 함부로 가로질러 지나가는
일을 막을 수 있습니다.

그런데 그 다른 로봇의 차례가 실제로 왔을 때는, 그 임시 선점을 풀어줘야 합니다
(그래야 그 로봇이 자기 목적지로 정상적으로 들어갈 수 있습니다). 이 "풀어주기"가
`unreserve(x, y, t)`입니다. 매번 테이블 전체를 다시 만드는 것보다, 딱 한 칸만
풀고 다시 `register_path`로 채우는 게 훨씬 빠릅니다 — 이건 성능 최적화이기도
합니다.

## 3.5 clear — 처음부터 다시 계획할 때

`clear()`는 테이블을 통째로 비웁니다. PBS가 `plan()`(처음부터 전체 계획)을
호출할 때마다 가장 먼저 하는 일입니다. 반면 `replan()`(선택적 재계획, 06장)에서는
테이블을 통째로 비우지 않고 "영향받지 않은 로봇들의 기존 경로"만 다시 채워 넣는
방식을 씁니다 — 이 차이가 선택적 재계획의 핵심 동작 중 하나입니다.

   나중에 어떤 기능에 필요한가? (→ 3.3)
