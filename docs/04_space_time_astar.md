# 4. Space-time A* — 로봇 한 대의 경로 찾기

[01~03장](01_mapf_problem.md)에서 준비한 재료(Cell, Map, ReservationTable)를 가지고
실제로 "로봇 한 대의 경로"를 찾는 엔진을 만듭니다. 이게 [01장 1.5절](01_mapf_problem.md)에서
말한 **저수준** 엔진입니다.

## 4.1 일반 A*와 무엇이 똑같고, 무엇이 다른가

같은 점: "지금까지 온 거리(g) + 목적지까지 남은 예상 거리(h)"가 가장 작은 상태를
먼저 펼치는 것은 동일합니다. 우선순위 큐(priority queue)를 쓰는 것도 같습니다.

다른 점은 딱 하나입니다 — **상태가 (x, y)가 아니라 (x, y, t)** 라는 것.
[02장 2.2절](02_map_and_agent.md)의 `SpaceTimeCell`이 바로 이 상태입니다.
그래서 "시공간(space-time) A*"라고 부릅니다.

## 4.2 탐색 과정 — 직관적으로

```
1. (start, t=0)에서 시작한다.
2. 매 단계마다, 현재 칸에서 "이동" 또는 "대기(wait)" 후보들을 모두 펼친다.
   - 이동: Map::neighbors()가 알려주는 상/하/좌/우 (벽이 아닌 칸만)
   - 대기: 제자리에 그대로, t만 +1
3. 각 후보 (x', y', t+1)에 대해:
   - ReservationTable에 이미 점유되어 있으면(Vertex conflict) → 건너뛴다.
   - 지금 칸 → 후보 칸으로의 이동이 Edge conflict라면 → 건너뛴다.
4. goal에 도착하면 경로를 반환한다.
5. 더 펼칠 후보가 없는데 goal에 못 도착했으면 "경로 없음".
```

3번이 일반 A*에는 없는, 시공간 A*만의 단계입니다. "다음 칸이 비어있는가"를
좌표만 보고 확인하는 게 아니라, **그 칸 + 그 시각**이 비어있는지를 확인합니다.
같은 칸이라도 시각이 다르면 비어있을 수 있습니다 — 다른 로봇이 그 칸을 "지나간 뒤"라면요.

## 4.3 휴리스틱 — 맨해튼 거리

A*가 "목적지까지 얼마나 남았는지" 추정하는 값이 휴리스틱(h)입니다. 여기서는
**맨해튼 거리** `|dx| + |dy|`를 씁니다 (격자에서 가로 이동 칸수 + 세로 이동 칸수).

이 값이 왜 안전한가: 맨해튼 거리는 실제 최단 이동 칸수보다 절대 더 크게
나오지 않습니다(벽이 있으면 실제로는 더 멀어질 수만 있지, 맨해튼 거리보다
짧아질 수는 없습니다). 이런 휴리스틱을 **admissible(과대평가하지 않음)** 이라
부르고, A*가 admissible한 h를 쓰면 반드시 최단 경로를 찾는다는 것이 보장됩니다.
([01장 1.1절](01_mapf_problem.md)에서 미리 언급한 내용입니다.)

## 4.4 max_timestep — 무한 탐색 방지

```cpp
struct AStarConfig {
    int max_timestep{256};
};
```

맵이 막혀 있거나 해가 아예 없는 경우, A*는 영원히 탐색을 계속할 수 있습니다.
t가 `max_timestep`을 넘으면 그 가지(branch)를 포기합니다 — 안전장치입니다.
작은 맵에서는 256이면 충분하지만, 맵이 아주 크거나 로봇이 많이 우회해야
하는 상황이라면 늘려야 할 수 있습니다.

## 4.5 search()의 반환값 — std::optional을 쓰는 이유

```cpp
std::optional<Path> search(Cell start, Cell goal, int start_time = 0);
```

"경로를 찾았거나, 못 찾았거나" 둘 중 하나입니다. 예외(exception)를 던지는
대신 `std::optional`을 쓰는 이유: 고수준 엔진(PBS, 05장)이 "이 로봇은 경로가
없다"는 상황을 **정상적인 흐름**으로 처리해야 하기 때문입니다. 예외를 쓰면
"실패"가 비정상적인 사건처럼 다뤄지지만, MAPF에서는 "막혀서 못 찾음"이
흔하게 일어나는 정상적인 결과의 일종입니다.

```cpp
if (auto path = astar.search(start, goal)) {
    // 경로 사용
} else {
    // 못 찾음 — 정상적인 분기로 처리
}
```

`start_time`이 0이 아닐 수도 있는 이유: 처음부터 계획할 때는 항상 0이지만,
**재계획**(06장)에서는 "이미 시간이 좀 흐른 상태에서, 지금부터 다시 계획"해야
합니다. 이때 `start_time`에 현재 시각을 넘기면, 반환되는 Path의 인덱스 0이
그 시각에 대응하도록 맞춰줍니다.

## 4.6 search_with_diagnostics() — "왜 막혔는지"까지 기록하는 버전

```cpp
struct AStarResult {
    std::optional<Path> path;
    std::vector<SpaceTimeCell> blocked_attempts;  // 실패했을 때만 의미 있음
};
```

사실 `search()`는 이 함수를 부르고 `.path`만 꺼내서 돌려주는 얇은 래퍼입니다.
탐색 로직은 한 곳에만 존재합니다(코드 중복 방지) — `search()`를 그대로 쓰는
입장에서는 아무것도 달라지지 않습니다.

이 "진단 버전"이 따로 필요한 이유: 일반 `search()`는 실패하면 그냥 `nullopt`만
주고 끝입니다. "어디서, 왜 막혔는지"에 대한 정보가 전혀 없습니다. 하지만
[06_selective_replan.md](06_selective_replan.md)의 **계층적 확장(Tiered Escalation)**
은 정확히 이 정보가 필요합니다 — "탐색하다가 점유 때문에 건너뛴 모든 시공간 칸"
목록(`blocked_attempts`)을 받아서, 각 칸의 주인을 `ReservationTable::get_owner()`로
조회하고, "누가 길을 막았는지"를 알아냅니다.

```
search()                  → 경로만 필요할 때 (대부분의 경우)
search_with_diagnostics() → 실패 원인까지 알아야 할 때 (06장의 계층적 확장)
```

## 4.7 정리 — 이 장에서 만든 것의 역할

```
[전략]   선택적 재계획                    (06장)
              ↓ 호출
[고수준] PBS                              (05장)
              ↓ 호출
[저수준] Space-time A*  ← 이 장에서 다룬 것
   "예약 테이블을 피해서, 로봇 한 대의 최단 경로를 찾아라"
```

이 엔진은 **로봇이 몇 대 있는지, 왜 재계획을 하는지 전혀 모릅니다.** 그냥
"여기부터 저기까지, 이 장부에 적힌 칸들은 피해서 최단 경로를 찾아라"라는
요청만 받고 응답합니다. 이렇게 책임을 좁게 한정한 게, 위층(PBS, 전략)을
간단하게 만드는 핵심입니다.
