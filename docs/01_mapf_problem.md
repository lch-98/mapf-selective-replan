# 1. MAPF란 무엇인가 — A*에서 출발하기

## 1.1 당신이 이미 아는 것: A*

A*는 **로봇 1대**가 출발점에서 목적지까지 가장 짧은 길을 찾는 알고리즘입니다.

- 매 단계마다 "지금까지 온 거리(g) + 목적지까지 남은 예상 거리(h)"가 가장 작은 칸을 먼저 펼칩니다.
- h(목적지까지 예상 거리)가 실제 거리보다 절대 크게 잡히지 않으면(이를 "admissible"이라 부릅니다),
  A*는 반드시 최단 경로를 찾아냅니다.

여기서 "공간(space)"만 다룹니다. 칸 (x, y)가 상태(state)입니다.

## 1.2 로봇이 여러 대가 되면 생기는 새로운 문제

로봇이 10대 있다고 합시다. "각 로봇마다 A*를 따로 돌려서 최단 경로를 구하면 되지 않을까?"

해보면 바로 문제가 생깁니다.

> 로봇 A의 최단 경로와 로봇 B의 최단 경로를 각각 따로 구했더니,
> 두 로봇이 **같은 시각에 같은 칸**을 차지하게 생겼다.

이게 바로 **충돌(conflict)** 입니다. 각자 따로 최적인 경로가, 합쳐놓으면 서로 부딪힙니다.
**MAPF (Multi-Agent Path Finding, 다중 에이전트 경로 탐색)** 는 "여러 로봇의 경로를
한꺼번에, 충돌 없이" 구하는 문제입니다.

## 1.3 충돌은 정확히 두 종류뿐이다

이걸 "대충 겹치면 안 된다"처럼 모호하게 정의하면 나중에 디버깅이 지옥이 됩니다.
정확히 두 가지로만 정의합니다.

### (1) Vertex conflict — 자리 충돌

같은 시각 t에, 두 로봇이 같은 칸 (x, y)에 동시에 존재.

```
시각 t:   로봇A → (3,3)
시각 t:   로봇B → (3,3)   ← 같은 칸, 같은 시각. 충돌.
```

### (2) Edge conflict — 자리 바꿈 충돌 (헤드온 충돌)

두 로봇이 동시에 자리를 맞바꾸려는 경우. 칸 자체는 같은 시각에 겹치지 않지만,
이동 중간에 정면으로 마주칩니다.

```
시각     t   →  t+1
로봇A: (0,0) → (1,0)
로봇B: (1,0) → (0,0)     ← 서로 마주보고 스쳐 지나가며 충돌
```

이 두 가지만 정확히 막으면 "충돌 없는 경로"가 보장됩니다. 이후 모든 코드
(예약 테이블, A*, PBS)는 전부 이 두 정의를 기준으로 동작합니다.

## 1.4 "시간"을 명시적으로 다뤄야 한다 — Space-time(시공간)

일반 A*는 "지금 (3,3)에 있다"만 압니다. 하지만 충돌을 판단하려면
"**언제** (3,3)에 있다"까지 알아야 합니다. 그래서 상태를 (x, y) 둘이 아니라
**(x, y, t)** 세 개로 확장합니다. 이걸 **시공간(space-time)** 이라 부릅니다.

```
일반 A*의 상태:         (x, y)
시공간 A*의 상태:       (x, y, t)
```

t는 이산적인 타임스텝입니다(0, 1, 2, ...). 매 타임스텝마다 로봇은
"한 칸 이동" 또는 "제자리에서 대기(wait)" 둘 중 하나를 합니다.
대기를 넣는 이유는 뒤에서([04_space_time_astar.md](04_space_time_astar.md)) 설명합니다 —
미리 말하면, "길이 막혔을 때 잠깐 기다린다"는 선택지가 없으면 풀 수 있는 문제도 못 풉니다.

## 1.5 MAPF를 푸는 알고리즘은 보통 "두 층"으로 나뉜다

MAPF를 푸는 방법은 다양하지만, 이 프로젝트(그리고 연구 커뮤니티 표준)는
문제를 두 층으로 쪼개서 풉니다.

```
[고수준]  여러 로봇이 서로 안 부딪히도록 "순서"나 "제약"을 조정하는 층
              ↓ 이 층이 저수준에게 "이 칸들은 피해서 찾아줘"라고 시킨다
[저수준]  로봇 한 대의 최단 경로를 찾는 층 (= 시공간 A*)
```

- **저수준**: "이미 정해진 제약(피해야 할 칸들) 안에서, 로봇 한 대의 최단 경로를 찾아라."
  → 이 프로젝트에서는 **Space-time A*** 가 담당합니다.
- **고수준**: "여러 로봇이 만들어내는 제약들을 어떻게 조율해서, 전체가 충돌 없이 풀리게 만들까?"
  → 이 프로젝트에서는 **PBS (Priority-Based Search)** 가 담당합니다.

이 두 층의 관계가 이해의 핵심입니다. 저수준은 "한 사람 길찾기"만 잘하면 됩니다.
고수준은 "한 사람 길찾기"를 여러 번, 적절한 순서로 호출해서 전체를 맞춥니다.
자세한 동작은 [04_space_time_astar.md](04_space_time_astar.md), [05_pbs.md](05_pbs.md)에서 다룹니다.

## 1.6 이 프로젝트는 거기에 "세 번째 층"을 하나 더 올린다

지금까지는 "한 번 깨끗하게, 정적인 상황에서 푸는" 이야기였습니다.
그런데 실제 물류창고에서는 로봇들이 이동하는 **도중에** 갑자기 장애물(다른 로봇,
사람, 떨어진 물건)이 나타납니다. 그러면 "다시 계산"을 해야 하는데:

- **제일 단순한 방법**: 장애물이 생기면 **로봇 전체**를 처음부터 다시 계획한다.
- **이 프로젝트가 제안하는 방법**: 장애물과 실제로 마주칠 **일부 로봇만** 다시 계획한다.

이게 바로 연구 제목의 "전체 경로 재계획 VS 선택적 경로 재계획"입니다.
이 부분이 정확히 무엇이고 왜 어려운지는 [06_selective_replan.md](06_selective_replan.md)에서
아주 자세히 다룹니다 — 거기가 이 연구의 핵심입니다. 지금 단계에서는
"두 층(저수준/고수준) 위에 세 번째 층(재계획 전략)이 하나 더 있다"는 구조만 기억하면 됩니다.

```
[전략]   선택적 재계획         ← 연구 기여, 06장에서 다룸
              ↓ 호출
[고수준]     PBS              ← 05장
              ↓ 호출
[저수준] Space-time A*        ← 04장
```
