# 6. 선택적 재계획 + 계층적 확장 — 이 연구의 핵심 기여

[01~05장](01_mapf_problem.md)에서 만든 두 엔진(저수준 Space-time A*, 고수준 PBS)은
**"한 번 깨끗하게, 정적인 상황에서"** 모든 로봇의 경로를 계획하는 도구입니다.
이 장에서 다루는 것은 다른 질문입니다.

> 로봇들이 이미 계획대로 움직이고 있는 **도중에**, 갑자기 장애물이 나타났다.
> 무엇을, 언제, 어디까지 다시 계산해야 할까?

이 질문에 대한 답이 바로 이 연구의 핵심 기여 — **선택적 재계획(Selective Replanning)
+ 계층적 확장(Tiered Escalation)** 입니다.

## 6.1 가장 단순한 대응: 전체 재계획

가장 쉬운 방법은 "장애물이 생기면 로봇 **전체**를 처음부터 다시 계획"하는 것입니다.
[05장](05_pbs.md)의 `PBS::plan()`을 그냥 다시 호출하면 됩니다(단, 출발점만
"현재 위치"로 바꿔서).

문제는 속도입니다. 로봇이 40대 있는데 장애물이 로봇 1대 경로에만 영향을 줬다고
해도, 전체 재계획은 장애물과 전혀 무관한 39대까지 처음부터 다시 탐색합니다.
실측(§6.7)으로 40대 기준 약 **20배** 느립니다.

## 6.2 선택적 재계획의 핵심 아이디어

> 장애물이 생겨도, 그 장애물과 **실제로 시공간적으로 마주칠** 로봇만
> 다시 계획하면 되지 않을까? 나머지 로봇은 원래 경로를 그대로 믿고 간다.

비유: 식당에서 한 손님의 자리가 갑자기 막히면, **그 손님만** 다른 빈자리로
옮겨준다. 다른 손님들은 자리를 옮길 필요가 없다.

## 6.3 RHCR과 무엇이 다른가 — "선택성의 축"

이 분야 표준 베이스라인인 RHCR(Li et al., 2021)도 "선택적으로" 다시 계획합니다.
하지만 선택하는 **기준(축)**이 다릅니다.

| | 선택성의 축 | 동작 |
|---|---|---|
| RHCR | 시간 축 | 모든 로봇을 재계획하지만, 앞쪽 `w`스텝만 충돌을 해소 |
| 이 프로젝트 | 에이전트(로봇) 축 | 영향받은 로봇만 재계획, 나머지는 손도 안 댐 |

### 6.3.1 두 축을 구체적인 숫자로 비교하기

로봇이 40대 있고, 각자 경로가 100스텝(t=0~99)짜리라고 합시다.
t=50 시점에 장애물이 나타나서 재계획이 필요해졌습니다. 이때 사실은
**두 개의 서로 다른 질문**이 있습니다.

1. **(누구를?)** 40대 중에서, 몇 대를 다시 계산할 것인가?
2. **(언제까지?)** 재계산하는 로봇의 경로 중, t=50부터 어디까지를 다시 계산할 것인가?

RHCR과 이 프로젝트는 이 두 질문에 정반대로 답합니다.

```
RHCR (시간 축 선택성):
  1번(누구를?)  → 선택 안 함. 40대 전부가 재계획 대상.
  2번(언제까지?) → 선택함. 앞으로 w스텝만(예: t=50~70). 그 뒤는 일단 안 건드리고,
                  시간이 지나면 또 "앞으로 w스텝"을 다시 계산한다(윈도우가 미래로 굴러간다).

  비유: 40명 전원에게 "다음 20분 일정만 다시 짜라"고 시킨다.
        장애물과 무관한 38명까지도 20분치는 매번 다시 계산해야 한다.

이 프로젝트 (에이전트 축 선택성):
  1번(누구를?)  → 선택함. 장애물과 실제로 마주치는 로봇만(예: 2대).
  2번(언제까지?) → 선택 안 함. 그 2대는 t=50부터 목적지(t=99)까지 통째로 다시 계산.

  비유: 40명 중 자리가 막힌 손님 2명만 골라서, 그 2명은 "식사 끝까지 전체 동선"을
        다시 짜준다. 나머지 38명은 손도 안 댄다.
```

| | 1번 (누구를?) | 2번 (언제까지?) |
|---|---|---|
| RHCR | 전부 (선택 안 함) | 앞으로 w스텝만 (선택함) |
| 이 프로젝트 | 영향받은 로봇만 (선택함) | 끝까지 전부 (선택 안 함) |

각자 **한쪽 질문에서만** "자른다"는 선택을 하고, 다른 쪽은 안 자릅니다.
이게 "두 축이 직교(독립적)"라는 말의 정확한 의미입니다 — 한쪽을 정하는 게
다른 쪽 선택에 영향을 주지 않습니다.

### 6.3.2 "합친다"는 것의 의미

두 선택이 독립적이라는 건, 이론적으로 **동시에 둘 다** 적용할 수 있다는
뜻입니다.

```
합친 버전: 영향받은 로봇만 골라서(에이전트 축으로 자르고),
           그 로봇들도 "끝까지"가 아니라 "앞으로 w스텝만"(시간 축으로도 자른다).

비유: 자리가 막힌 손님 2명만 골라서(에이전트 축),
      그 2명한테도 "식사 끝까지 전부"가 아니라 "다음 20분치만" 다시 짜달라고 한다.
      두 가지 절약을 동시에 적용하는 것.
```

지금 이 프로젝트는 "합친 버전"을 만들지 않고, **에이전트 축 하나만 깊게
파서**(Tier 0/1/2 계층적 확장까지 만들어서) 연구 기여로 삼고 있습니다.
"합치는 것"은 지금 구현 대상이 아니라, 이 연구가 끝난 뒤 확장할 수 있는
미래 가능성으로만 언급한 것입니다.

## 6.4 Tier 0 — 가장 단순한 선택적 재계획

`PBS::replan()`이 받는 입력은 다음 네 가지입니다.

```cpp
std::optional<ReplanResult> replan(
    const std::vector<Agent>& agents,        // 전체 로봇 목록 (우선순위 = 벡터 순서)
    const PBSResult& previous_paths,         // 직전까지의 전체 경로
    const std::vector<Cell>& new_obstacles,  // 새로 생긴 장애물 칸들
    int current_time);                       // 재계획을 시작하는 절대 시각
```

동작 (코드의 1단계, `replan()` 앞부분):

```
1. "영향받는 로봇" 판별:
   각 로봇의 기존 경로(previous_paths)를 보고,
   current_time 이후 시점에 new_obstacles 중 한 칸과 겹치는지 확인한다.
   → 겹치면 "영향받음"으로 표시. 겹치지 않으면 그 로봇은 절대 안 건드린다.

   (구현 디테일: 로봇이 이미 목적지에 도착해서 거기 계속 머무는 경우
    — 05장 5.5절의 Tail — 도, current_time이 경로 길이를 넘었으면
    "목적지에 계속 있다"고 보고 같은 방식으로 검사한다.)

2. 영향받는 로봇만, "현재 위치"(current_time 시점에 있던 칸)에서
   "원래 목적지"까지 다시 SpaceTimeAStar로 탐색한다.
   이때 영향받지 않은 로봇들의 기존 경로는 예약 테이블에 그대로
   등록해서 "피해야 할 장벽"으로 쓴다.

3. 영향받는 로봇 전원이 새 경로를 찾으면 성공.
   "과거 구간(이미 지나온 길, 0~current_time-1)" + "새로 찾은 미래 구간"을
   이어붙여서 반환한다.
```

이게 `try_replan_set()` 함수가 하는 일입니다(working_ids = 영향받은 로봇 목록을
받아서, 그 로봇들만 agents 순서대로 — 즉 우선순위 순서대로 — 재탐색합니다).
이때 "영향받지 않은 로봇"의 기존 경로도 테이블에 등록해야 한다는 점이
중요합니다 — 안 그러면 재탐색되는 로봇이 가만히 있는 다른 로봇을 뚫고
지나갈 수 있습니다.

## 6.5 Tier 0의 한계 — "비켜주면 풀리는데, 못 비키는" 문제

다음 상황을 생각해 보세요.

> 영향받은 로봇 R이 장애물을 피해 우회하려 한다. 그런데 그 우회로에
> **다른 로봇 K가 마침 그 시간에 지나가고 있어서** 우회조차 막힌다.
> K는 장애물과 전혀 안 겹쳤으므로 Tier 0의 "영향받는 로봇" 판정에서
> 빠진다 — 그래서 K는 절대로 다시 계획되지 않는다.
> 결과: R은 새 경로를 못 찾고, Tier 0 전체가 실패로 끝난다.

하지만 실제로는 **"K가 한두 걸음만 비켜주면"** 풀리는 경우가 많습니다.
Tier 0는 "영향받지 않은 로봇은 절대 안 건드린다"는 원칙 때문에 이 가능성을
스스로 차단하고 있습니다.

이게 추상적인 우려가 아니라 실측으로 확인된 문제입니다 — 무작위 시나리오
실험(`benchmark_replan.py`)에서, 선택적 재계획(Tier 0만)의 성공률이 전체
재계획보다 낮게 나오는 경우가 있었습니다(로봇 20대 시나리오에서 84~86%
vs 86~90%). "선택적으로 하면 빠르지만 가끔 못 푸는 문제가 있다"가 실제로
관찰된 것입니다.

## 6.6 Tier 1 — 계층적 확장: "막은 원인을 찾아서 같이 풀어준다"

해결 아이디어: Tier 0가 실패했을 때, **무작위로 아무나 더 끼워넣는 게 아니라**,
"정확히 누가 길을 막았는지"를 결정론적으로 추적해서 그 로봇(K)까지 재계획
대상에 새로 포함시키고 다시 시도합니다.

이걸 가능하게 하는 것이 앞 장들에서 미리 깔아둔 두 가지 장치입니다 — 이 시점에서
왜 그렇게 설계했는지가 전부 연결됩니다.

- [03장 3.3절](03_reservation_table.md): ReservationTable이 "찼다/안 찼다"가 아니라
  **"누가 찼는지"**까지 기억하도록 만든 이유가 바로 여기서 쓰입니다.
- [04장 4.6절](04_space_time_astar.md): `search_with_diagnostics()`가 "막혀서
  못 간 시공간 칸들"(`blocked_attempts`)을 반환하도록 만든 이유도 여기서 쓰입니다.

코드 흐름(`try_replan_set`이 실패를 반환할 때):

```
1. Tier 0에서 막혀서 못 간 칸들의 목록(blocked_attempts)을 가져온다.
2. 그 칸들 각각에 대해 ReservationTable::get_owner()로 "누가 거기 있었는지" 조회한다.
3. 조회된 로봇 id들을 "구조 로봇 후보"로 모은다(중복 제거, owner == -1은 제외 —
   -1은 "주인 모름"이라 추적할 대상이 아니다. 03장 3.3절 참고).
4. 그 후보들을 원래 "영향받은 로봇" 집합(working_ids)에 합쳐서,
   처음부터 다시 한 번 agents 순서대로(우선순위 순서 그대로) 재탐색한다.
5. 성공하면 끝. 또 실패하면 "또 막은 원인"을 다시 추적해서 한 단계 더 확장한다
   (PBSConfig.max_escalation_tiers로 몇 단계까지 허용할지 설정 가능, 기본값 1).
```

`replan()`의 메인 루프(코드의 2단계)는 정확히 이 절차를 반복합니다 —
`tier`를 0부터 `max_escalation_tiers`까지 늘려가며 `try_replan_set()`을
다시 호출하고, 실패할 때마다 `newly_blocked_owners`를 `working_ids`에
추가합니다. 더 추가할 후보가 없으면(`added_any == false`) 그 자리에서
루프를 멈추고 바로 다음 단계(전체 재계획 폴백)로 넘어갑니다.

### 6.6.1 tier가 늘어날 때 정확히 무슨 일이 일어나는가 (`max_escalation_tiers = 2` 예시)

여기서 헷갈리기 쉬운 점이 있습니다 — "Tier 1이 실패하면 Tier 1을 한 번 더
반복한다"가 **아닙니다.** 정확히는 "영향받은 로봇 집합(working_ids)을 한
단계 더 넓혀서, **그 넓어진 집합 전체를 처음부터** 다시 한 판 돈다"입니다.
`max_escalation_tiers = 2`로 설정했을 때를 단계별로 따라가 보겠습니다.

```
tier=0 (Tier 0):
  working_ids = [R]                      ← R: 장애물과 직접 겹쳐서 영향받은 로봇
  try_replan_set([R]) 시도
  → 실패. R이 가려던 칸을 K가 막고 있었다(get_owner로 확인).
  → working_ids에 K 추가 → working_ids = [R, K]

tier=1 (Tier 1):
  working_ids = [R, K]                   ← R과 K를 "처음부터" 다시 agents 순서로 재탐색
  try_replan_set([R, K]) 시도
  → 또 실패. 이번엔 K가 가려던 칸을 M이 막고 있었다(R이 아니라 K가 새로 부딫힌 것).
  → working_ids에 M 추가 → working_ids = [R, K, M]

tier=2 (Tier 2, max_escalation_tiers=2라서 여기까지 허용됨):
  working_ids = [R, K, M]                ← 셋을 "처음부터" 다시 agents 순서로 재탐색
  try_replan_set([R, K, M]) 시도
  → 성공하면: escalation_tier = 2, rescued_ids = [K, M] 로 반환하고 끝.
  → 또 실패하면: tier가 max_escalation_tiers(2)에 도달했으므로 루프 종료
                 → 안전망(전체 재계획)으로 폴백.
```

즉 매 tier마다 일어나는 일은 "이전 tier의 실패 원인이 된 **새 로봇 한 명**을
working_ids에 추가하고, **누적된 working_ids 전체를 처음부터 통째로** 다시
계획"입니다. "R만 따로", "K만 따로" 재탐색하는 게 아니라, 항상 working_ids에
있는 로봇 전부를 agents 순서(우선순위)대로 한 번에 재탐색합니다 — 이래야
[05장](05_pbs.md)의 "먼저 계획된 로봇의 경로는 절대 안 바뀐다"는 PBS 규칙이
working_ids 안에서도 그대로 유지됩니다.

새로 추가할 원인 로봇이 더 없는 경우(같은 로봇이 또 막혔거나, owner가
-1=주인 모름인 경우)에는 `added_any == false`가 되어 더 높은 tier를
시도하지 않고 바로 안전망으로 넘어갑니다 — 어차피 같은 결과가 반복될
뿐이므로 헛수고를 하지 않습니다.

### 아주 중요한 구현 디테일 — "구조 로봇"의 과거 경로를 장벽으로 쓰면 안 된다

Tier 1에서 새로 끌고 온 K(구조 로봇)를 재탐색할 때, **아직 K의 차례가 오지
않은 시점**에는 K의 "원래(과거) 경로"를 장벽으로 등록하면 안 됩니다.

왜 안 되는가: K를 데려온 이유가 "K가 비켜줄 기회를 주기 위해서"입니다.
그런데 K의 옛 경로를 그대로 장벽으로 써버리면, K는 "여전히 원래 자리에
있는 것처럼" 취급되어서 결국 아무것도 안 비킨 것과 같아집니다 — 데려온
의미가 없어집니다.

`try_replan_set`의 1단계/2단계 구조가 이걸 정확히 구현합니다. agents를
순회하면서 working_set(working_ids)에 속한 로봇은 "아직 처리 안 된 동안은
장벽에서 제외"하고(자기 차례가 와서 새로 재탐색되고 `register_path`로
등록될 때만 장벽이 됩니다), working_set에 없는 로봇만 처음부터 기존 경로
그대로 장벽으로 등록합니다. 이게 [05장 5.2절](05_pbs.md)의 `PBS::plan()`이
"아직 계획 안 된 낮은 우선순위 로봇은 장부에 없다"고 보는 것과 정확히
같은 원리입니다 — Tier 1은 "구조 로봇도 결국 agents 순서(우선순위) 안에서
정상적으로 한 번 더 계획되는 로봇일 뿐"이라는 관점으로 처리됩니다.

### 막힌 원인을 추적하는 두 번째 경로 — "barrier 등록 자체가 거절되는" 경우

지금까지 설명한 Tier 1 추적(`blocked_attempts` → `get_owner`)은 **working_ids
로봇의 A* 탐색이 막혔을 때**를 다룹니다. 그런데 실제로 구현하고 테스트해보니,
A*가 멀쩡히 경로를 찾았는데도 그 다음 단계인 **`register_path`(barrier 등록)
자체가 거절되는 경우**가 따로 있었습니다 — 이건 `blocked_attempts`로는 전혀
잡히지 않는 별개의 실패 경로입니다.

```
상황: working_ids = {R}만 있다고 하자(K는 아직 안 들어와 있다).
1. R(working_ids)이 A*로 새 경로를 찾고 register_path로 등록 — 성공.
2. K(working_ids에 없음)를 "장벽"으로 등록하려는데, K의 *옛 경로*에 들어있는
   어느 (칸,시각)이 R의 *새* 경로(또는 R의 Tail)와 정확히 겹친다.
   → register_path(K, K의 옛 경로)가 false를 반환.
```

이 경우 `blocked_attempts`는 비어 있습니다(A* 탐색 자체는 실패한 적이 없으니까).
그래서 `get_owner`로 추적할 대상이 없습니다 — 사실 추적해야 할 것은 "그 칸의
주인"이 아니라 **"barrier 등록을 거절당한 로봇(K) 자신"**입니다. K가
working_ids에 없어서 아직 비켜줄 기회를 못 받은 게 원인이기 때문입니다.

그래서 `try_replan_set`은 이 경우를 위해 별도 채널(`out_blocked_owner`)로
"거절당한 로봇의 id"를 직접 알려줍니다. `replan()`의 확장 단계는 이 값이
있으면 그 로봇을 곧바로 working_ids에 추가합니다 — `blocked_attempts` 기반
추적과 동시에 동작하는, 사실상 "Tier 1의 두 번째 입구"입니다.

## 6.7 Tier 2 — 안전망: 그래도 안 되면 전체 재계획

Tier 1을 `max_escalation_tiers`까지 확장해도 안 풀리면(또는 더 추가할 원인
로봇이 없으면), 마지막 안전망으로 **전체 재계획**(`full_replan_fallback`)으로
넘어갑니다. 단, "처음부터(t=0, 원래 출발점)"가 아니라 **"지금 각 로봇이 있는
위치"를 새 출발점**으로 삼아서 다시 계획합니다 — 이미 지나온 길은 버리지
않습니다.

`ReplanResult`에는 이번 재계획이 어디까지 갔는지 알 수 있는 정보가 들어있습니다.

```cpp
struct ReplanResult {
    PBSResult        paths;           // 최종 전체 경로 (영향 안 받은 로봇은 기존 경로 그대로)
    std::vector<int> replanned_ids;   // 실제로 다시 탐색된 로봇 id 전부
    int escalation_tier{0};           // 0=Tier 0에서 끝남, 1+=그 단계까지 확장해서 성공, -1=전체 재계획 폴백
    std::vector<int> rescued_ids{};   // Tier 1+에서 새로 추가된 "구조 로봇" id 목록
};
```

`escalation_tier`와 `rescued_ids`는 알고리즘 동작 자체에는 영향을 주지 않는
**순수 디버깅·연구용 정보**입니다 — "이번 재계획이 얼마나 선택적이었는지"를
측정하는 지표로 씁니다.

## 6.8 전체 흐름 한 장 요약

```
장애물 발생
   ↓
[1] 영향받는 로봇 판별 (current_time 이후 new_obstacles와 겹치는 경로)
   ↓
Tier 0: 영향받은 로봇만 재탐색
   │ 성공 → 끝 (escalation_tier = 0)
   │ 실패 ↓
Tier 1: 막힌 칸의 주인(owner) 로봇을 추적해서 working_ids에 추가, 재탐색
   │ 성공 → 끝 (escalation_tier = 1, rescued_ids = [그 로봇])
   │ 실패, 추가할 후보 있음 ↓
Tier 2..N: (max_escalation_tiers까지) 같은 방식으로 반복 확장
   │ 그래도 실패, 또는 더 추가할 후보 없음 ↓
안전망: 전체 재계획 (현재 위치 기준, 과거 이동분은 보존)
   → 끝 (escalation_tier = -1)
```
