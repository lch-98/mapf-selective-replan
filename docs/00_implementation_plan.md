# 0. 구현 계획 — 무엇을, 어떤 순서로, 어떻게 검증하며 만들 것인가

[01~07장](01_mapf_problem.md)에서 "왜 이렇게 설계했는지"를 다졌습니다.
이제부터는 **코드**를 만듭니다. 이 장은 코드를 짜기 전에 먼저 "어떤 순서로,
어떤 도구로, 어떻게 검증하며 만들 것인가"를 정하는 계획서입니다. 계획 없이
코드를 짜기 시작하면 중간에 길을 잃기 쉽습니다. 번호를 0번으로 둔 이유는
이 문서가 — [00_index 역할도 겸해서 — 구현을 시작하기 전에 가장 먼저 펴봐야
하는 지도이기 때문입니다.

## 0.1 기본 원칙

1. **docs의 장 순서 = 코드 구현 순서.** 01~06장을 만든 순서 그대로,
   가장 작은 단위(Cell)부터 가장 큰 단위(PBS::replan)까지 쌓아 올립니다.
2. **컴파일되는 가장 작은 단위부터.** 한꺼번에 여러 파일을 작성하지 않고,
   "이거 하나만 추가하면 빌드+테스트가 통과한다"는 단위로 잘라서 진행합니다.
3. **모든 구현 단계는 테스트를 동반합니다.** 코드를 작성한 직후, 그 코드가
   docs에서 설명한 대로 동작하는지 확인하는 작은 테스트를 같이 만들고
   실행합니다. "일단 다 만들고 나중에 테스트"는 하지 않습니다.
4. **이전 폴더(`Path_Planning_Core`, 언더스코어 없음)는 참고하지 않습니다.**
   핵심 아이디어(엔진 2개 + 전략 1개, Tier 0/1/2)는 유지하지만, 코드와
   그 안의 논리는 이번에 직접 처음부터 다시 검증합니다.

## 0.2 구현 순서 (= docs 장 순서)

| 순서 | 만드는 것 | 대응 문서 | 파일 |
|---|---|---|---|
| 0 | 빌드 시스템 뼈대 (그릇만) | — | `CMakeLists.txt` (최상위/core/tests) |
| 1 | `Cell`, `SpaceTimeCell`, `Map`, `Agent` | [02장](02_map_and_agent.md) | `core/include/mapf/{map,agent}.hpp`, `core/src/map.cpp` |
| 2 | `ReservationTable` | [03장](03_reservation_table.md) | `core/include/mapf/reservation_table.hpp`, `core/src/reservation_table.cpp` |
| 3 | `SpaceTimeAStar` | [04장](04_space_time_astar.md) | `core/include/mapf/space_time_astar.hpp`, `core/src/space_time_astar.cpp` |
| 4 | `PBS::plan` | [05장](05_pbs.md) | `core/include/mapf/pbs.hpp`, `core/src/pbs.cpp` (plan만) |

각 단계는 "그 전 단계가 테스트까지 통과한 뒤에만" 시작합니다. 예를 들어
1번(Map)의 테스트가 실패하는 상태에서 2번(ReservationTable)을 만들기
시작하지 않습니다 — 토대가 흔들리면 그 위에 쌓는 모든 것이 같이 흔들립니다.

## 0.3 각 단계의 표준 절차

```
[1] 지도 확인  — 지금 docs/0X_*.md의 어느 절을 코드화하는지 짚는다
[2] 설계       — 자료구조/함수의 시그니처(입력·출력)를 먼저 말로/표로 정한다
[3] 헤더 작성  — .hpp에 "무엇을 할 수 있는가"를 선언 (주석으로 왜 그런지도 적음)
[4] 구현 작성  — .cpp에 "어떻게 하는가"를 작성
[5] 테스트 작성 — 그 기능이 docs에서 말한 대로 동작하는지 확인하는 테스트
[6] 빌드+실행  — 실제로 컴파일하고 테스트를 돌려서 결과를 직접 보여줌
[7] 해석       — 통과/실패 각 줄을 풀어서 설명
     (실패 시) [디버깅] 가설 → 확인 → 수정 → 재실행, 전 과정을 같이 진행
```

## 0.4 테스트 도구 — GoogleTest

C++ 코드가 맞게 동작하는지 자동으로 확인하려면 "테스트 프레임워크"가 필요합니다.
이 프로젝트는 **GoogleTest**를 씁니다. 이유:

- C++ 생태계에서 가장 널리 쓰이는 표준 테스트 도구라서 자료가 많습니다.
- CMake의 `FetchContent` 기능으로 "설치"할 필요 없이 빌드 시점에 자동으로
  받아올 수 있습니다 — Windows든 Ubuntu든 똑같은 `CMakeLists.txt`로 동작합니다
  (0.6절에서 자세히 설명).

테스트 코드의 모양을 미리 보면:

```cpp
#include <gtest/gtest.h>
#include "mapf/map.hpp"

TEST(MapTest, EmptyMapIsAllPassable) {
    mapf::Map map(3, 3);
    EXPECT_TRUE(map.is_passable(1, 1));
}
```

`TEST(그룹이름, 테스트이름)` 하나가 "확인하고 싶은 사실 하나"입니다.
`EXPECT_TRUE(조건)`은 "이 조건이 참이어야 한다"는 뜻이고, 거짓이면 그 줄에서
실패가 기록되지만 같은 테스트의 나머지 줄은 계속 실행됩니다(반면 `ASSERT_TRUE`는
실패하면 그 테스트를 즉시 중단합니다 — 상황에 맞게 골라 씁니다).

## 0.5 빌드 도구 — 컴파일러는 MSVC(Visual Studio 2022)로 확정

처음에는 MinGW-w64(g++)도 검토했습니다 — Ubuntu의 g++와 거의 동일한
컴파일러라서 호환성 확신이 더 크다는 장점이 있었습니다. 하지만 실제로
이 컴퓨터에 설치된 컴파일러를 확인해보니:

```
PATH에 잡혀 있던 g++  → 6.3.0 (2016년 버전) — C++17도 제대로 지원 안 함, 이 프로젝트는
                          std::optional 등 C++17 기능을 핵심적으로 쓰므로 부적합
이미 설치된 MSVC       → 14.43 (Visual Studio 2022) — 최신, C++20/23까지 잘 지원, 즉시 사용 가능
```

최신 MinGW-w64를 새로 설치하는 것보다, 이미 설치되어 있고 최신인 **MSVC를
바로 쓰는 것**으로 확정했습니다.

**Ubuntu 이전과 충돌하지 않는 이유**: [0.6절](#06-cmake란-무엇인가--처음-보는-경우를-위한-설명)에서
보듯, CMake는 "어떤 컴파일러를 쓸지"를 코드가 아니라 빌드 설정으로 분리합니다.
지금 Windows에서는 `CMakeLists.txt`가 MSVC에게 빌드를 시키고, 나중에 Ubuntu로
넘어가면 같은 `CMakeLists.txt`가 g++(또는 clang)에게 빌드를 시킵니다 — 코드
자체(.hpp, .cpp)는 그대로 두고 컴파일러 지정만 바뀝니다. 단, **MSVC 전용
문법(비표준 확장)을 쓰지 않고 표준 C++만 쓴다**는 규칙은 계속 지켜야 합니다 —
이건 [09_windows_to_ubuntu.md](09_windows_to_ubuntu.md)에서 구체적인 체크리스트로
다룹니다.

## 0.6 CMake란 무엇인가 — 처음 보는 경우를 위한 설명

CMake는 "컴파일러에게 무엇을 어떻게 빌드하라고 시킬지"를 적는 설정 언어입니다.
직접 컴파일러 명령어(`cl.exe /c map.cpp ...` 또는 `g++ -c map.cpp ...`)를 손으로
치는 대신, `CMakeLists.txt` 파일에 "이 폴더의 .cpp 파일들을 모아서 라이브러리로
만들어라" 같은 규칙을 적어두면, CMake가 **현재 운영체제·컴파일러에 맞는** 빌드
명령을 자동으로 생성합니다.

```
CMakeLists.txt (사람이 작성, OS·컴파일러 무관)
        ↓ CMake가 읽고
Windows + MSVC:   Visual Studio 프로젝트(.sln) 생성
Ubuntu  + g++:    Makefile 생성
```

같은 `CMakeLists.txt` 한 벌로 양쪽 OS·컴파일러에서 빌드할 수 있다는 게
핵심입니다 — 이게 사용자님이 요청한 4번 조건(Windows→Ubuntu 호환)을
만족시키는 실질적 방법입니다.
