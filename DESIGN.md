# 다중 모바일 로봇 실시간 경로 재계획 — 설계 문서

> 물류 환경에서의 동적 장애물을 고려한 다중 모바일 로봇 실시간 경로 재계획 · 설계 레퍼런스 (v1)

이 문서는 엔진 코드 개발을 시작하기 전에 합의한 설계 결정을 고정해 두는 용도입니다.
개발 중 결정이 바뀌면 이 문서를 함께 갱신하세요(살아있는 문서).

---

## 0. 프로젝트의 두 층위

| 구분 | 목적 | 범위 |
|------|------|------|
| **데모 버전** (지금) | 실력 + 연구 방향을 보여주는 증거 | 아래 로드맵의 일부만 떼어낸 축소판 (2D) |
| **연구 주제** (향후) | 실제 논문으로 이어질 본 연구 | 전체 로드맵 (3D Gazebo, 정량 평가 포함) |

데모와 논문은 **같은 C++ 코어 위에** 세운다. 그래야 일관성이 생기고,
이후에는 코어에 ROS2 래퍼만 붙이면 자연스럽게 확장된다.

## 1. 핵심 설계 원칙 — 단일 코어 + 두 어댑터

> **알고리즘 C++ 코어는 GUI도 ROS도 모르게 만든다.**
> 맵·로봇·장애물을 넣으면 경로를 뱉는 순수 라이브러리로만 만들고,
> 얇은 어댑터 두 개만 붙인다.

```
                 ┌─────────────────────────────────┐
                 │       C++ 알고리즘 코어          │
                 │ (프레임워크 비의존, 단일 소스)    │
                 │  Space-time A* / SIPP           │
                 │  PBS (+ CBS 비교군)              │
                 │  선택적 재계획 전략              │
                 └───────────────┬─────────────────┘
                  ┌──────────────┴───────────────┐
          ┌───────▼────────┐            ┌────────▼────────┐
          │ pybind11 바인딩 │           │ ROS2 노드 래퍼   │
          │ (C++ → Python) │            │ (C++ → ROS2)    │
          └───────┬────────┘            └────────┬────────┘
          ┌───────▼──────────┐          ┌────────▼────────┐
          │ Python 2D GUI    │          │   Gazebo 3D     │
          │ pygame·인터랙티브 │          │ Ubuntu·다중 로봇│
          └──────────────────┘          └─────────────────┘
```

알고리즘은 **딱 한 번만** 작성한다. 2D 프로토타입에서 검증한 그 코어가
그대로 Gazebo로 올라간다. 두 군데에 알고리즘을 흩뿌리지 않는다.

## 2. 알고리즘은 3계층이다 — 엔진 vs 전략

가장 중요한 개념 정리. **"선택적 재계획"은 space-time A* + PBS와 같은 층위가 아니다.**
이 둘은 *엔진*이고, 선택적 재계획은 그 엔진을 *언제·누구에게 돌릴지* 결정하는 한 층 위의 *전략*이다.

```
┌──────────────────────────────────────────────┐
│  [전략]  재계획 전략  ← 연구의 핵심 기여       │   언제·누구를 다시 푸나
│          선택적 재계획 + 계층적 확장           │
│          (PBS::replan, Tiered Escalation)    │
└───────────────────┬──────────────────────────┘
                    │ 호출(calls)
┌───────────────────▼───────────────────────────┐
│  [엔진]  고수준 조정                           │   한 번 푸는 방법
│          PBS (구현됨) · CBS(향후 비교군)       │
└───────────────────┬───────────────────────────┘
                    │ 호출(calls)
┌───────────────────▼───────────────────────────┐
│  [엔진]  저수준 탐색                           │
│          Space-time A* (구현됨) · SIPP(논문용) │
└───────────────────────────────────────────────┘
```

- **저수준 탐색**: 단일 로봇 1대의 경로. 다른 로봇/장애물을 예약 테이블·제약으로 받아 회피.
  `space-time A*` 또는 `SIPP`.
- **고수준 조정**: 로봇들이 서로 충돌하지 않게 조정. `PP`(고정 우선순위) → `PBS`(우선순위 탐색) → `CBS/ECBS`(최적/유계 준최적).
- **재계획 전략**(= 내가 직접 짜는 부분, 연구 기여): "동적 장애물 등장 → 영향받은 로봇 식별 →
  걔네만 우선순위 순으로 엔진 재호출, 그래도 막히면 원인 로봇까지 단계적으로 확장". 이 로직
  자체가 *선택적 재계획 + 계층적 확장*이고, 엔진을 *포함하지만* 엔진과 같지 않다.
  자세한 동작은 §5.3에서 예시와 함께 설명한다.

## 3. 엔진 선택 — 데모 → 논문 업그레이드 경로

| 계층 | 데모 (지금) | 논문 (향후) |
|------|-------------|----------------|
| 저수준 | `space-time A*` (구현 쉬움) | **`SIPP`** 로 교체 (동적 장애물 전용, 더 빠름) |
| 고수준 | `PBS` | `PBS` 유지 + `CBS/ECBS`를 비교군으로 추가 |
| 전략 | 선택적 재계획 (동일) | 선택적 재계획 (동일) — 전략 층은 안 바뀐다 |

- **데모에선 space-time A*로 시작**한다. 구현이 단순해 빠르게 동작을 보여줄 수 있다.
- **논문 단계에서 저수준만 SIPP로 교체**한다. SIPP는 궤적이 알려진 동적 장애물 사이에서
  (위치 × 시간) 공간을 "안전 구간(safe interval)"으로 압축해 탐색하므로, 동적 물류 환경에 더 적합하다.
  SIPP는 우선순위 기반 계획과 그대로 결합된다 → **SIPP + PBS**.
- **전략 층은 2D·3D·데모·논문에서 모두 동일**하게 유지된다. 이게 단일 코어 설계의 핵심 이득.

## 4. 연구 포지셔닝 — 기존 연구와의 관계 및 차별점

### 4.1 RHCR와의 관계 — "선택성의 축"이 다르다

내 도메인(대규모 창고 물류)의 표준 베이스라인은 **RHCR (Li et al., 2021)**.
여기서 "선택성"의 *축*이 다르다는 점이 논문의 1차 차별점이다.

| | 선택성의 축 | 동작 |
|---|---|---|
| **RHCR** | 시간 축 | 모든 로봇 재계획, 단 앞쪽 `w` 스텝만 충돌 해소 (`h`마다 재계획) |
| **내 제안 (선택적 재계획)** | 에이전트 축 | 영향받은 로봇만 재계획 |

두 축은 직교 → **조합 가능**(윈도우 안에서 영향받은 로봇만 재계획).

### 4.2 부분 재계획·계층적 확장 계열 선행 연구와의 관계

"에이전트 축으로 재계획 범위를 제한한다"는 발상 자체는 이미 여러 갈래로 존재한다.
**이 발상을 독창적인 것으로 주장하면 안 된다** — 대신 "기존 부분 재계획 기법들이 공유하는
약점을, PBS의 우선순위 구조를 그대로 활용해서 보완한다"가 정확한 포지셔닝이다.

| 선행 연구 | 핵심 아이디어 | 이 프로젝트와의 관계 |
|---|---|---|
| **PBS** (Ma et al., 2019) | 새 우선순위 제약이 생기면 그보다 낮은 우선순위 에이전트만 재탐색 | 이 프로젝트가 사용하는 고수준 엔진 자체. "낮은 우선순위만 재탐색"이라는 PBS 본연의 선택성을 동적 장애물 재계획에 그대로 적용한 것이 Tier 0 |
| **Svancara et al., 2019 "Online MAPF" (PIEF)** | 새 목표가 생긴 에이전트와 그로 인해 막힌 에이전트만 재계획, 나머지는 고정 | Tier 0와 거의 동일한 발상. 다만 "그래도 막히면 어떻게 하는가"는 다루지 않음 — 이 공백이 이 프로젝트의 출발점 |
| **MAPF-LNS / LNS2** (Large Neighborhood Search) | 충돌하는 에이전트의 부분집합을 무작위/적응적으로 골라 재계획, 나머지는 고정 | "부분집합만 재계획"이라는 틀은 같지만, 부분집합을 **무작위/학습된 휴리스틱**으로 고름 |
| **IC-MAPF**, **WinC-MAPF** | 막히면 단계적으로(yield → 정적 재계획 → 동적 재계획 → 공동계획) 대응 범위를 넓힘 | "단계적 확장"이라는 구조는 같지만 별도의 윈도우/정보량 개념을 새로 도입함 |

### 4.3 이 프로젝트의 정확한 차별점 — "계층적 확장(Tiered Escalation)"

기존 부분 재계획 기법들의 공통된 약점: **"영향받지 않은 에이전트의 경로를 절대 건드리지
않는다"는 제약 때문에, 영향받은 에이전트가 좁은 우회 공간에서 막히면 그냥 실패한다.**
실제로는 "그 우회로를 막고 있는 로봇 한 대가 비켜주면" 풀릴 수 있는 상황인데도, 부분 재계획은
그 가능성을 원천적으로 차단한다 (자세한 실측 증거는 §5.3 참고).

이 프로젝트는 이 약점을 다음 방식으로 보완한다 — **PBS가 이미 가진 우선순위 구조 안에서,
무작위가 아니라 "A* 탐색이 실패한 원인이 된 로봇을 결정론적으로 추적"해서 재계획 범위를
점진적으로 넓힌다.** 이게 LNS(무작위/적응적 부분집합)나 IC-MAPF/WinC-MAPF(별도의
윈도우·정보량 개념)와 다른, 이 프로젝트만의 구체적인 기여다.

```
Tier 0: 장애물과 직접 시공간적으로 겹치는 로봇만 재계획 (기존 PBS/PIEF와 동일한 선택성)
   ↓ 실패하면
Tier 1: A* 탐색이 "막혀서 못 간 시공간 칸"의 주인(owner) 로봇을 추적해서
        그 로봇까지 재계획 대상에 포함 → 다시 시도 (결정론적 확장, 무작위 아님)
   ↓ 그래도 실패하면 (설정값 max_escalation_tiers까지 반복 확장 가능)
Tier 2: 안전망으로 전체 재계획 (현재 위치 기준, 과거 이동분은 보존)
```

**논문 구도:**
- 베이스라인: ① 전체 재계획, ② RHCR(시간 윈도우 선택성), ③ Tier 0만 쓰는 단순 선택적 재계획
- 제안 기법: 계층적 확장(Tiered Escalation)을 적용한 에이전트 선택적 재계획
- 엔진: `(SIPP +) PBS` (공통, §3 참고)
- 측정: 재계획 시간/성공률/재계획된 로봇 수 + **"Tier 1이 실제로 구원한 비율"**
  (Tier 0만으론 실패했는데 Tier 1로는 성공한 비율) — 이 지표가 계층적 확장의 기여를 직접 증명

핵심 메시지: "엔진을 새로 만든다"가 아니라 **"검증된 엔진(PBS) 위에서, 기존 부분 재계획
기법들의 공통된 실패 지점을 결정론적 원인 추적으로 보완하는 재계획 전략을 제안한다."**
→ 석사 범위에 적절하고 방어하기 좋다.

## 5. 구현 로드맵

### Phase 0 — 문제 정형화 (3~5일)
- 월드 모델: 격자(4- 또는 8-연결), 이산 타임스텝, 한 칸 = 로봇 1대
- 충돌 정의: vertex conflict(같은 시각 같은 칸), edge conflict(자리 바꿈) — **정확히 못 잡으면 디버깅 지옥**
- 동적 장애물: "예측 가능(등속 직선)"부터. 나중에 "예측 불가"로 확장
- 맵 포맷: 처음부터 MovingAI MAPF 벤치마크 포맷을 따른다 (재현성/반박 방지)

### Phase 1 — C++ 코어: 정적 MAPF
- 저수준 `space-time A*`: (x, y, **t**) 시공간 격자 A*. 예약 테이블 입력 받아 단일 로봇 최단 경로
- 고수준: **PBS** 구현. CBS는 비교군으로 나중에
- GoogleTest 단위 테스트("충돌 없음", "경로 유효"), CMake 빌드
- 끝점: "맵+에이전트 → 충돌 없는 경로 집합"이 콘솔에서 검증됨 — **자세한 동작 원리는 §5.2 참고**

### 5.2 코어 동작 원리 — "엔진" 두 층을 처음부터 끝까지

이 절은 §2의 [엔진] 두 층(저수준 탐색, 고수준 조정)이 **실제로 코드에서 어떻게 동작하는지**를
한 줄씩 풀어서 설명한다. C++ 경험이 적어도 따라올 수 있도록 비유를 쓴다.

#### 5.2.1 비유로 먼저 잡는 큰 그림

물류창고에 로봇이 여러 대 있다고 생각하자. 이 코어는 "식당 예약 시스템"과 똑같이 동작한다.

- **칸(Cell)** = 식당의 테이블 자리
- **시각(t)** = 예약 시간대 (1번째 턴, 2번째 턴, ...)
- **예약 테이블(ReservationTable)** = "어느 자리가 몇 시에 누구한테 예약되어 있는지" 적어둔 장부
- **저수준 탐색(SpaceTimeAStar)** = "장부를 보고, 비어있는 자리·시간만 골라서 한 사람의 동선을 짜는 사람"
- **고수준 조정(PBS)** = "손님들에게 번호표를 나눠주고, 1번부터 차례로 동선을 짜게 만들고 짠 동선을 장부에 적는 관리자"

핵심은: **장부(예약 테이블)에 먼저 적힌 사람의 동선은 절대 안 바뀐다.** 나중 번호표를 받은 사람은
이미 적힌 동선을 피해서 자기 동선을 짜야 한다. 이게 PBS가 충돌을 막는 방식의 전부다.

#### 5.2.2 자료구조: Cell, Path, ReservationTable

| 타입 | 정의 | 비유 |
|---|---|---|
| `Cell{x, y}` | 격자 위의 한 칸 좌표 | 테이블 자리 번호 |
| `Path` (`= vector<Cell>`) | 한 로봇의 경로. `path[t]`가 "t번째 시각에 있는 칸" | 한 사람의 전체 동선표 |
| `ReservationTable` | `(x, y, t) -> 그 자리를 예약한 로봇 id` 를 담은 장부 | 예약 장부 |

`ReservationTable`은 두 종류의 충돌을 막는다:
- **Vertex conflict(자리 충돌)**: 같은 시각에 같은 칸에 두 로봇이 있으면 안 됨 → `reserve(x,y,t,agent_id)`로 등록, `is_occupied(x,y,t)`로 확인
- **Edge conflict(자리 바꿈 충돌)**: 로봇 A가 (0,0)→(1,0)으로 가는 동시에 로봇 B가 (1,0)→(0,0)으로 가면 중간에서 부딪힘 → `reserve_edge(...)`로 양방향 모두 차단

(2026-06-22 변경) `ReservationTable`은 원래 "이 칸이 비었나/찼나"만 기억했는데(`unordered_set`),
지금은 **"누가 찼는지"까지** 기억한다(`unordered_map<자리, 로봇id>`). 이유는 §5.3에서
"누가 길을 막고 있는지"를 추적해야 하기 때문 — 예약 장부에 "예약자 이름"까지 적어두기 시작한 것.

#### 5.2.3 저수준 엔진: SpaceTimeAStar — "한 사람의 동선 짜기"

`SpaceTimeAStar::search(start, goal, start_time)`는 **딱 한 로봇**의 최단 경로를 찾는다.
보통 A* 알고리즘과 똑같은데, 칸(x,y)뿐 아니라 **시각(t)도 같이** 따라다니는 게 다른 점이다 — 그래서
"시공간(space-time) A*"라고 부른다.

```
탐색 과정 (직관):
  1. (start, t=0)에서 시작.
  2. 매 단계마다 "다음 칸으로 이동" 또는 "제자리에서 대기(wait)" 둘 다 시도.
  3. 다음 후보 칸이 ReservationTable에 이미 예약되어 있으면(다른 로봇이 그 시각에 있음) 건너뛴다.
  4. goal에 도착하면 끝. 큐가 다 빌 때까지 못 도착하면 "경로 없음".
```

(2026-06-22 변경) `search()`는 이제 내부적으로 `search_with_diagnostics()`라는 "더 자세히 기록하는
버전"을 부르고 그 결과 중 경로만 꺼내서 돌려준다. 진단 버전은 **실패했을 때 "어디서 막혔는지"도
같이 기록**한다 — 이게 §5.3의 계층적 확장이 쓰는 정보다. (기존 코드를 쓰는 입장에서는 아무것도
달라지지 않는다 — `search()`를 그대로 호출하면 똑같이 동작한다.)

#### 5.2.4 고수준 엔진: PBS — "여러 사람의 동선을 충돌 없이 짜기"

`PBS::plan(agents)`는 **여러 로봇** 전체의 경로를 한 번에 짠다. 핵심 규칙은 단 하나:

> **agents 리스트에서 앞에 있는 로봇일수록 우선순위가 높다. 높은 순위가 먼저 동선을 짜고,
> 그 동선을 장부에 적는다. 낮은 순위는 장부에 적힌 모든 동선을 피해서 자기 동선을 짠다.**

```
PBS::plan() 동작:
  1. 예약 테이블을 깨끗이 비운다.
  2. agents[0](1순위)부터 차례로:
     a. SpaceTimeAStar로 "지금 장부 상태"를 보고 이 로봇의 경로를 찾는다.
     b. 못 찾으면 → 전체 실패(nullopt 반환).
     c. 찾으면 → 그 경로를 장부에 적는다(register_path). 이후 모든 낮은 순위 로봇이 이걸 피한다.
  3. 전원 성공하면 "로봇id → 경로" 표를 반환.
```

추가로 "아직 차례가 안 된 로봇이 목적지에 이미 도착해서 작업 중인 다른 로봇의 자리를 멋대로
지나가는" 문제를 막기 위해, 각 로봇을 탐색하기 직전에 "다른 모든 로봇의 목적지"를 임시로
선점해 둔다 — "그 자리는 누군가의 최종 도착지니까 함부로 지나가면 안 돼"라는 규칙.

### 5.3 전략층 동작 원리 — "선택적 재계획 + 계층적 확장"을 처음부터 끝까지

**이 절이 논문의 핵심 기여를 설명하는 부분이다.** §5.2의 엔진(PBS, A*)은 "한 번 깨끗하게
계획하는" 도구다. 전략층은 "이미 계획이 다 끝나서 로봇들이 움직이고 있는 도중에, 갑자기
장애물이 생겼을 때 *무엇을, 언제, 어디까지* 다시 계산할지"를 결정한다. `PBS::replan()` 함수
하나가 이 전략을 전부 구현한다 (`core/src/pbs.cpp`).

#### 5.3.1 왜 "전체 재계획"이 아니라 "선택적 재계획"인가

가장 단순한 대응은 "장애물 생기면 로봇 전체를 처음부터 다시 계획"하는 것이다(이게 Phase 2까지의
방식, §4.3의 "베이스라인 ①"). 문제는 로봇 수가 늘어날수록 느려진다 — 장애물과 전혀 무관한
로봇까지 매번 처음부터 다시 탐색하기 때문이다. 실측(§5.3.6)으로 40대 로봇 기준 약 **20배** 차이가
난다.

**선택적 재계획의 핵심 아이디어**: "장애물이 생겨도, 그 장애물과 *실제로 시공간적으로 마주칠*
로봇만 다시 계획하면 되지 않을까?" 나머지 로봇은 원래 경로를 그대로 믿고 가던 길을 계속 간다.

#### 5.3.2 Tier 0 — 가장 단순한 선택적 재계획

```
PBS::replan(agents, previous_paths, new_obstacles, current_time):

  1. "영향받는 로봇" 판별:
     각 로봇의 기존 경로(previous_paths)가, current_time 이후 시점에
     new_obstacles 중 한 칸과 겹치면 "영향받음" 표시.
     → 겹치지 않으면 그 로봇은 원래 경로 그대로 유지(절대 안 건드림).

  2. 영향받는 로봇만, "현재 위치"(current_time 시점의 칸)에서 "원래 목적지"까지
     다시 SpaceTimeAStar로 탐색. 이때 영향받지 않은 로봇들의 경로는
     예약 테이블에 그대로 등록해서 "피해야 할 장벽"으로 사용.

  3. 영향받는 로봇 전원이 새 경로를 찾으면 성공.
     "과거 구간(이미 지나온 길)" + "새로 찾은 미래 구간"을 이어붙여서 반환.
```

비유: 식당에서 한 손님의 자리가 갑자기 막히면, **그 손님만** 다른 빈자리를 찾아서 옮겨준다.
다른 손님들은 자리를 옮길 필요가 없으니 그대로 둔다.

#### 5.3.3 Tier 0의 한계 — "비켜주면 풀리는데 못 비키는" 문제

문제 상황: 영향받은 로봇이 우회하려는데, 그 우회로에 **다른 로봇이 마침 그 시간에 지나가고
있어서** 우회조차 막힌다고 하자. 그 다른 로봇은 "장애물과 안 겹쳤으니" Tier 0에서는 절대로
다시 계획되지 않는다 — 그래서 Tier 0는 그냥 "실패"로 끝나버린다.

하지만 실제로는 **"그 다른 로봇이 한두 걸음만 비켜주면"** 풀리는 경우가 많다. Tier 0는 "절대
안 건드린다"는 원칙 때문에 이 가능성을 스스로 차단하고 있다. 이게 §4.3에서 설명한 "기존 부분
재계획 기법들의 공통된 약점"이고, 실제로 무작위 실험(`benchmark_replan.py`)에서 선택적
재계획의 성공률이 전체 재계획보다 낮게 나오는 경우가 확인됐다 (예: 로봇 20대 시나리오에서
84~86% vs 86~90%).

#### 5.3.4 Tier 1 — 계층적 확장: "막은 원인을 찾아서 같이 풀어준다"

해결 아이디어: Tier 0가 실패했을 때, **"정확히 누가 길을 막았는지"를 추적**해서 그 로봇까지
재계획 대상에 새로 포함시키고 다시 시도한다. 무작위로 아무나 더 끼워넣는 게 아니라, A* 탐색이
실패한 *진짜 원인*을 결정론적으로 짚어내는 것이 핵심이다.

이걸 가능하게 하려고 두 가지를 바꿨다(§5.2.2, §5.2.3에 나온 변경):
- `ReservationTable`이 "이 칸이 찼다"뿐 아니라 **"누가 찼다"**까지 기억하도록 확장 (`get_owner(x,y,t)`)
- `SpaceTimeAStar`가 탐색 실패 시 **"막혀서 못 간 시공간 칸들"**의 목록을 같이 반환하도록
  `search_with_diagnostics()`를 추가

```
Tier 1 동작 (실패한 Tier 0 시도에 이어서):
  1. Tier 0에서 막혀서 못 간 칸들의 목록(blocked_attempts)을 가져온다.
  2. 그 칸들 각각에 대해 ReservationTable.get_owner()로 "누가 거기 있었는지" 조회.
  3. 조회된 로봇 id들을 "구조 로봇 후보"로 모은다(중복 제거).
  4. 그 후보들을 원래 "영향받은 로봇" 집합에 합쳐서, 처음부터 다시 한 번 순서대로 재탐색
     (PBS의 "우선순위 = 리스트 순서" 규칙을 그대로 따름 — 새 규칙을 만들지 않음).
  5. 이번엔 성공하면 끝. 또 실패하면 "또 막은 원인"을 다시 추적해서 한 단계 더 확장
     (PBSConfig.max_escalation_tiers로 몇 단계까지 허용할지 설정 가능, 기본값 1).
```

**아주 중요한 구현 디테일** (실제로 처음 구현했을 때 버그였던 부분): "구조 로봇"으로 새로
추가된 로봇을 재탐색할 때, **아직 자기 차례가 오지 않은 시점**에는 그 로봇의 "원래(과거) 경로"를
장벽으로 등록하면 안 된다. 그렇게 등록하면 "비켜줄 기회를 주려고 데려왔는데, 여전히 원래 자리에
있는 것처럼" 취급되어 버려서 확장한 의미가 없어진다. 그래서 PBS는 "이번 시도에서 이미 처리된
로봇"만 장벽으로 쓰고, 아직 처리 안 된(구조 대상으로 새로 들어온) 로봇은 장벽에서 빼둔다 — 이게
`plan()`이 "아직 계획 안 된 낮은 순위 로봇은 장부에 없다"고 보는 것과 같은 원리다.

#### 5.3.5 Tier 2 — 안전망: 그래도 안 되면 전체 재계획

Tier 1을 몇 단계까지 확장해도 안 풀리면(또는 더 추가할 원인 로봇이 없으면), 마지막 안전망으로
**전체 재계획**으로 넘어간다. 단, "처음부터"가 아니라 **"지금 각 로봇이 있는 위치"를 새 출발점**으로
삼아서 다시 계획한다 — 이미 지나온 길은 버리지 않는다.

`ReplanResult`에는 이번 재계획이 어디까지 갔는지 알 수 있는 정보가 들어있다:
- `escalation_tier`: 0=Tier 0에서 끝남, 1 이상=그 단계까지 확장해서 성공, -1=전체 재계획 폴백
- `rescued_ids`: 계층적 확장으로 새로 추가된 "구조 로봇" id 목록 (비어있으면 Tier 0만으로 충분했다는 뜻)

#### 5.3.6 실측 결과 — 무엇을 확인했고, 무엇이 남았는가

`python/benchmark_replan.py`로 같은 맵·같은 초기 경로·같은 장애물 시나리오에 대해 (A) 전체
재계획, (B) Tier 0만 쓰는 선택적 재계획, (C) Tier 0+1(계층적 확장)을 비교했다.

**확인된 것:**
- 속도: 선택적 재계획(B, C)은 로봇 수가 늘수록 전체 재계획(A)보다 압도적으로 빠르다
  (40대 기준 약 20배). 계층적 확장을 켜도 이 속도 우위는 거의 그대로 유지된다.
- 정확성: GoogleTest로 "좁은 통로에서 로봇 1대가 길게 막고 있어서 Tier 0는 실패하는데, Tier 1은
  그 로봇을 정확히 찾아내 비켜주게 만들어서 성공하는" 시나리오를 직접 설계해서 검증함
  (`tests/test_phase1.cpp`의 `PBSEscalationTest` 스위트). 의도한 대로 동작한다.

**아직 부족한 것 (정확도/실효성 개선 과제):**
- 지금 쓰는 데모 맵(16×16, 빈 공간 위주)에서 무작위 시나리오로 돌려보면, "Tier 0는 실패했는데
  Tier 1로 구원되는" 경우가 매우 드물게만 나타난다(로봇 40대 기준 50번 중 1번, 2%). 맵이 넓어서
  "다른 우회로가 이미 있는" 경우가 많아 Tier 1이 나설 차례가 잘 안 생기기 때문으로 보인다.
- 다음 검토 과제: ① 통로가 좁고 복잡한(물류 창고형) 맵에서 재실험해 Tier 1 활용 빈도가 실제로
  올라가는지 확인, ② Tier 1이 찾아주는 경로의 "품질"(최적성, 우회로 길이)을 전체 재계획과 비교,
  ③ `max_escalation_tiers`를 2 이상으로 늘렸을 때의 트레이드오프(선택성이 희석되는 지점) 측정.

  Harmonic(신버전)을 쓰려면 `ros_gz` 브릿지 패키지를 non-default 조합으로
  수동 설정해야 해서 이 프로젝트 범위에서는 불필요한 리스크.
- Gazebo Classic 11 자체는 2025년에 EOL 예정이지만, 이 프로젝트의 Phase 4
  작업 일정(3~4주)에는 지장이 없다. 이후 신버전으로 옮길 필요가 생기면
  이 섹션만 갱신하면 됨(코어 알고리즘 코드는 `core/`에 있어 영향 없음).
- **Harmonic 전환 비용은 낮다고 판단**: MAPF 알고리즘(`core/`)은 Gazebo와
  완전히 분리되어 있어(§8 "C++ vs Python" 원칙과 동일하게, 시뮬레이터는
  순수 백엔드) 전환 시 건드릴 대상은 `rclcpp` 래퍼 노드의 launch 파일,
  월드 SDF, 플러그인 이름(`gazebo_ros_*` → `ros_gz_*`), 토픽 브릿지 설정
  정도. 알고리즘 로직 재작성은 필요 없다.

## 7. 평가 지표

- 성공률 (success rate)
- makespan
- sum-of-costs
- **재계획 1회 소요 시간 (ms)** ← 핵심 주장 근거
- **재계획된 로봇 수** ← 선택적 재계획의 우위 근거
- **Tier 1이 추가로 구원한 비율** (Tier 0만으론 실패했는데 Tier 0+1로는 성공한 비율) ←
  계층적 확장의 기여를 직접 증명하는 지표 (§5.3.6 참고, `benchmark_replan.py`에서 자동 집계)
- **Tier 1 발동 비율** (escalation_tier == 1인 시도의 비율) ← 맵 구조에 따라 계층적 확장이
  얼마나 자주 "쓰일 일이 있는지"를 보여주는 지표

## 8. 꼭 피해야 할 함정

1. **격자(이산) ↔ Gazebo(연속)의 간극** — 셀↔월드 좌표 변환 레이어 +
   "다음 웨이포인트 도착 시 다음 스텝 진행" 동기화 로직 필수.
2. **Gazebo 다중 로봇 네임스페이싱** — 토픽/TF 충돌. Phase 4 전에 "TurtleBot3 2대 충돌 없이 텔레옵"부터 검증.
3. **C++ vs Python** — 코어는 C++로 직행. 단, Phase 3 *재계획 트리거 로직* 한 조각만은
   파이썬 노트북에서 먼저 검증 후 C++로 옮기면 디버깅 시간 절약.
